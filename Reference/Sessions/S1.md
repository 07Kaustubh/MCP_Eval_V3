## Procedure

1. **Read the inputs above.** Internalize the persona's voice from the brief and the V3 reference prompts. Internalize the levers selected in `Hardness_Plan.md`.

2. **Draft `5_Prompt.txt`.** Plain prose, persona's voice, 500-word cap, no em-dashes, no tool names, no MCP-server names, no internal IDs, no pre-solving. Engineer the selected hardness levers into the situation — each lever should appear naturally as part of the persona's framing, not as an obvious test hook.

3. **Run the validator (cheap gate).**
   ```
   python Validators/validate.py --phase prompt --task Tasks/<TASK_DIR>
   ```
   Read `_aux/Validator_Reports/prompt.md`. If FAIL, fix every fail, re-run until clean.

4. **Spawn Council A — Grounding.** Follow `Reference/Council_Protocol.md`. The sub-agent extracts every concrete claim in `5_Prompt.txt` and verifies it against `_aux/Universe_Split/`. Output goes to `_aux/Council_Reports/S1_A_grounding.md`. Block on any ungrounded claim.

5. **Spawn Council B — Adversarial QC.** Follow `Reference/Council_Protocol.md`. Scores every applicable QC sub-dim from `Docs/7_QC_Spec_Doc1.json` against the prompt, attempts a second valid reading that flips a write action, and runs the adversarial sweep. Output goes to `_aux/Council_Reports/S1_B_adversarial.md`. Block on any Major / Moderate issue.

6. **Loop.** If either council blocks, apply fixes, re-run the validator, re-run BOTH councils. Loop until clean.

7. **Run the similarity gate against the prior-task corpus.** Both councils have cleared the prompt on quality; this gate checks it does not collide with prior submissions before we let it through to the platform linter (where a similarity hit costs the operator a round trip).
   ```
   python Validators/calc_similarity.py Tasks/<TASK_DIR>
   ```
   The script writes `_aux/Similarity_Report.json` with the top matches against every prior `5_Prompt.txt` in `Tasks/` and the V3 reference prompts. Read the report.

   | Score band | Action |
   |---|---|
   | < 40 | PASS. Proceed to step 8. The composite is multiplicatively weighted by contextual differentiators (persona / business function / universe) — a sub-40 score means the prompt is either genuinely lexically distinct OR has strong contextual differentiators that would justify invalidation at S1.5 if the platform linter ever flagged it. Log the top match + composite to `_aux/Reasoning/prompt_design.md` for the record. |
   | ≥ 40 | STOP. PIVOT mandatory — the composite is high EVEN AFTER contextual weighting, meaning constants align with a prior task and the structural similarity is genuine (not just surface lexical overlap). Do NOT proceed to step 8. Re-draft the prompt using `Reference/Similarity_Pivot.md` (pick the smallest combination of the 6 levers that breaks the match while preserving the Hardness Plan levers), then loop back to step 3 (validator) + step 4 + step 5 + step 7 against the fresh draft. Log the pivot to `Tasks/_meta/Similarity_Log.md` with the date, top-match reference, composite + per-dim scores, and the resulting similarity score. |

8. **Strict veteran audit (MANDATORY, auto-fire).** Spawn an audit sub-agent (`oracle`) using the prompt template in `Reference/Sessions/AUDIT.md` with `--phase prompt` configured. Save the report to `_aux/Council_Reports/AUDIT_prompt.md`. AUDIT applies the STRICTEST QC interpretation — 5/5 is the only acceptable score on every applicable sub-dim (no NON-FAIL band justifications), density bar is 50+ midpoint (not 40 floor), every "should" reads as "must". This catches what Council A + B + similarity gate missed.

   | Verdict | Action |
   |---|---|
   | `PASS (STRICT)` | Proceed to step 9 (final report). |
   | `REVISE` | Apply the audit's per-issue fixes IN PLACE in `5_Prompt.txt`. Re-run step 3 (validator) + step 4 (Council A) + step 5 (Council B) + step 7 (similarity gate) + step 8 (AUDIT) against the revised prompt. Iteration cap: **3 REVISE rounds**. After 3, end your response with the full audit issue list and escalate to the user. |
   | `REBUILD` | The prompt has structural issues a fix-in-place cannot repair (e.g., the prompt's framing actively blocks 3+ Hardness levers, second valid reading flips multiple write actions). End your response with the audit's REBUILD recommendation and recommend `PIPELINE REDO — Tasks/<TASK_DIR>` in a fresh chat. |

9. **Final report.** Append a short verdict to `_aux/Reasoning/prompt_design.md` summarizing: which levers were engineered into the prompt, expected stump targets, council verdicts, the final similarity score with top match, and the AUDIT verdict.

## Exit criteria

- `5_Prompt.txt` exists, word count ≤ 500, no em-dashes.
- Validator returns PASS for the prompt phase.
- Council A returns GO with zero ungrounded claims.
- Council B returns GO with every applicable QC sub-dim at 5 (or any 3-4 explicitly justified against per-task universe).
- Council B-B3 returns projected tool-call count ≥ 50 (design target) OR 40-49 with explicit `THIN_DENSITY` justification carried from HARDNESS. < 40 BLOCKS.
- Council B-B4 returns "all Hardness levers from `_aux/Hardness_Plan.md` still triggered".
- Similarity gate returns composite < 40 against the prior-task corpus (`_aux/Similarity_Report.json` present, `max_composite < 40`). Composite ≥ 40 STOPs to PIVOT.
- Strict veteran audit returns `PASS (STRICT)` (`_aux/Council_Reports/AUDIT_prompt.md` present with that verdict). `REVISE` iterations must be resolved within the 3-round cap; `REBUILD` STOPs to REDO.

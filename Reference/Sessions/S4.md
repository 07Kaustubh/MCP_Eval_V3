# PIPELINE S4 — Verifier-Fails Analysis

Triggered by: `PIPELINE S4 — Tasks/<TASK_DIR>` followed by paste of verifier fails + trajectories already deposited into `Agent_Responses/`.

## Phase-readiness gate (run FIRST, before anything else)

```
python Validators/phase_ready.py --phase s4 --task Tasks/<TASK_DIR>
python Validators/parse_trajectories.py Tasks/<TASK_DIR>
```

The first script refuses to proceed if 5/6/7 + 8_Verifier_Fails.txt aren't present.
The second parses real trajectories and writes `_aux/Trajectory_Stats.json` with measured `pass@1` + `avg_tool_calls_total`. This is the empirical ground for the verdict — no manual JSON parsing.

If `parse_trajectories.py` returns `REBUILD_CANDIDATE_DENSITY` or `REBUILD_CANDIDATE_DIFFICULTY`, S4 cannot save the task — the user must invoke `PIPELINE REDO`.

## What this phase does

For each rubric that failed across runs, classify the failure: rubric-invalid, judge-error, or legitimate model failure. Produce AF justifications for legitimate failures and fix-lists for invalid rubrics or judge errors.

## Procedure

0.5. **Trajectory hard gates (v12 T2 + T3 — run BEFORE classification).**

   **T3 — Error Rate Gate:** Count the number of runs in `Agent_Responses/` that errored out (did not complete to a verifier-evaluable state). If the count is **≥ 3**, the task FAILs the Error Rate sub-dim per QC spec — at most 2 erroneous runs are allowed (need at least 4 successful runs). Record in `_aux/Council_Reports/S4_verdict.md`:
   ```
   ## Trajectory T3 — Error Rate
   Erroneous runs: <count>/6. Verdict: PASS (< 3) | FAIL (>= 3).
   ```
   On FAIL, STOP and tell the user — the task cannot be evaluated meaningfully; re-run trajectories.

   **T2 — Agent Failure Rate Gate (pass@1 ≤ 40%):** Across the SUCCESSFUL (non-error) runs, count how many passed ALL valid rubrics. The pass rate = passed_runs / completed_runs must be ≤ 40% (i.e., for 6 runs with 0 errors, at most 2 may pass everything; for 4 successful + 2 errors, at most 1 may pass everything). Record:
   ```
   ## Trajectory T2 — Agent Failure Rate
   Runs passing all rubrics: <count>/<completed>. pass@1: <pct>%. Verdict: PASS (≤ 40%) | FAIL (> 40%).
   ```
   On FAIL, the task is "Too Easy" per QC spec — STOP and route to `PIPELINE REDO` (the task needs harder levers).

1. **Build the rubric x run matrix.** For each rubric in `7_Rubrics.json` and each run in `Agent_Responses/`, mark pass / fail / not-evaluated. Identify rubrics that failed all completed runs (AF rubrics) and rubrics that failed some runs (partial fails).

2. **For EVERY failing rubric, walk the trajectory BEFORE classifying.** Trajectory analysis is what distinguishes the three buckets — without it, you cannot tell a judge error from a legitimate fail or from a rubric-invalid case. Skipping the walk and classifying from the verifier text alone is the most common S4 defect.

   Per failing rubric, build a trajectory walk record covering ALL 6 runs (or however many completed):
   - **Relevant tool calls.** Every tool call across the runs that touches the rubric's subject (e.g., for "Agent sends email to robert.hayward@..." → list every `email_send` call in every run with recipient, subject, body excerpt).
   - **Parameter values.** Exact parameters the agent used (recipient, subject, body content, IDs, amounts, dates).
   - **Tool response.** Did the call succeed? What did the tool return? Was the artifact created with the expected values?
   - **Rubric criterion comparison.** Walk the rubric's criterion text and decide whether the trajectory evidence satisfies it — with the rubric's flexibility convention applied ("approximately" tolerates rounding; "(or similar)" tolerates phrasing variants; exact-match values must match exactly).

   ONLY AFTER the trajectory walk, classify into ONE of three buckets. **Every bucket entry MUST carry a trajectory citation in the form `Run X, tool call Y: <parameter values>` (or `Run X: action not attempted` for absence-of-action) — not just Bucket 2.**

   ### Bucket 1 — Rubric Invalid
   The rubric is the problem. Signs:
   - Outcome too strict (expects exact value, agent reported equivalent rounded value).
   - Outcome bundles two independent facts that pass / fail separately.
   - Process rubric punishes a valid alternative path.
   - Process rubric covers a property an Outcome already proves.
   - Phrasing fails a valid agent expression (e.g., expects exact phrase, accepts no synonyms).
   Trajectory citation required: name the trajectory step that produced the valid alternative the rubric did not accept (e.g., `Run 2, tool call 14: agent sent email with subject 'rate-lock review' — rubric expected exact phrase 'rate lock expiration review'`).
   Action: write the fix to `_aux/Council_Reports/S4_fixes.md` with a before / after, the trajectory citation, and reasoning.

   ### Bucket 2 — Judge Error
   The rubric is correct. The agent's trajectory shows it actually satisfied the rubric. The judge misread. Action: log to `_aux/Council_Reports/S4_judge_errors.md` with the trajectory citation (run #, tool call #, parameter value) proving the agent satisfied the rubric.

   ### Bucket 3 — Legitimate Model Failure (AF Justification)
   The rubric is correct. The agent really did fail. This is desired difficulty. Trajectory citation required: name the trajectory step where the agent went wrong (e.g., `Run 3, tool call 8: agent reported gross $186,350 instead of net $117,000`) OR `Run X: action not attempted` if the agent never tried the action.

   **Pre-write 5-point checklist (v15) — ask all 5 BEFORE writing the AF justification.** If any answer is NO, the rubric is the problem (Bucket 1), not the model. Reclassify and fix the rubric, do not write an AF justification.

   1. Is the criterion self-contained, atomic, and grounded in the universe's ground truth? (Judge can evaluate without external lookup; one independent claim; values exist in `_aux/Universe_Split/`.)
   2. Is it flexible enough to allow valid alternative approaches without unfairly penalizing the agent? (E.g., if prompt says "get Emeka up to speed", Slack is just as valid as email.)
   3. Is the criterion actually required by the prompt, rather than asking for something extra? (Re-read the prompt; trace the criterion back to a specific ask.)
   4. Does it use real tool names and valid parameters? (Verify against the universe-correct tool catalog (Brookfield: `Brookfield_Base_Universe/8_Server_Tools_Details.json`; KeyStone: `Mortgage_Base_Universe/6_Server_Tools_Details.json`); `search_crm` is not real, `crm_search_contacts` is.)
   5. Could a capable agent realistically pass this task? (If 0 of 6 runs passed despite a valid approach, the rubric may be unachievably strict.)

   If all 5 = YES → proceed to Bucket 3, write the AF justification.
   If any = NO → the rubric is the problem, not the model. Reclassify as Bucket 1 (Rubric Invalid) and add a fix to `_aux/Council_Reports/S4_fixes.md`.

   **Reference examples** (from the QC docs — patterns to emulate / avoid):

   GOOD AF justifications (cite the concrete failure mode):
   - "The agent consistently failed to identify the correct coordinator responsible for the DOT gap. This reflects a real reasoning failure that required cross-service inference."
   - "All runs sent the email to noah.fitzgerald@greenstack.com instead of noah.fitzpatrick@axiom.com. This entity confusion is the intended failure mode for the task."

   BAD patterns (these reveal a rubric issue and require a reclassification):
   - "Agent never used search_crm tool" → `search_crm` is not a real tool; rubric must be updated to use the real tool name (`crm_search_contacts`).
   - "Agent never included the exact phrase X" → if the same idea has multiple valid expressions, the rubric is over-specific; rewrite with `(or similar)`.
   - "Agent notified Emeka via Slack instead of email" → if the prompt only said "get Emeka up to speed", Slack is valid; the rubric improperly locks in a channel.
   - "Agent didn't update all 9 tickets" → first verify ground truth requires all 9; if so, write one atomic rubric per ticket rather than bundling.

   Action: write the AF justification using the template in `Reference/Linter_Playbook.md` — two to four sentences, no em-dashes, no references to guides / specs / frameworks, first-person, cites the concrete data fact and the specific reasoning gap. Save to `_aux/Council_Reports/S4_AF_justifications.md`.

   **All-Failing Rubrics sub-dim scoring (mandatory per v11):** after all failing rubrics are classified, compute the Bucket 1 ratio = `Bucket 1 count / total failing rubrics count`. Apply the threshold:

   | Bucket 1 ratio | All-Failing Rubrics sub-dim score | Verdict |
   |---|---|---|
   | > 50% | **1/5 (FAIL)** | Most all-failing rubrics are caused by invalid rubric design, not by genuine task difficulty. The rubric set itself is the problem. |
   | 25-50% | **3/5 (NON-FAIL)** | A meaningful fraction of failures trace to rubric issues; investigate the rubric quality. |
   | < 25% | **5/5 (PASS)** | Most failures are legitimate model failures or judge errors. The rubric set is sound. |

   Record the score in `_aux/Council_Reports/S4_verdict.md` under a new `## All-Failing Rubrics sub-dim` section with the Bucket 1 ratio + the assigned score + brief justification citing the failing rubrics involved.

   This closes the Rubrics-Eval "All-Failing Rubrics sub-dim defers to verifier stage" gap — the verifier stage IS S4, and v11 makes the scoring explicit instead of leaving the sub-dim permanently at "N/A → 5".

3. **Confirm classifications against the universe.** For every Bucket 1 / Bucket 2 classification, re-grep `_aux/Universe_Split/` to confirm the rubric value vs the agent's trajectory value. Council A handles this if any classification is uncertain.

4. **Update the Hardness Plan calibration.** Compare the stump hypothesis predictions in `_aux/Hardness_Plan.md` against the actual AF rubrics:

   - Not predicted + failed → under-predicted (which lever did we miss?).
   Append the calibration delta to `Tasks/_meta/Hardness_Patterns_Log.md`.

- `Tasks/_meta/Hardness_Patterns_Log.md` updated.

# PIPELINE S1.5 — Linter Blocker

Triggered by: `PIPELINE S1.5 — Tasks/<TASK_DIR>` followed by the platform linter output pasted below the line.

## What this phase does

Handles platform linter blocks on the prompt. Two classes:

- **Class A** — misalignment on a specific QC dimension. Decide: revise the prompt, or push back with a concise justification.
- **Class B** — similarity ≥ 40% with a prior submission. Pivot the prompt using `Reference/Similarity_Pivot.md`. No justification path.

## Required inputs

| File | Source |
|---|---|
| `Tasks/<TASK_DIR>/5_Prompt.txt` | S1 |
| `Tasks/<TASK_DIR>/_aux/Universe_Split/*` | S0 |
| `Tasks/<TASK_DIR>/_aux/Universe_Index/*` | S0 |
| `Tasks/<TASK_DIR>/_aux/Hardness_Plan.md` | HARDNESS |
| `Reference/Linter_Playbook.md` | decision flow + justification template |
| `Reference/Similarity_Pivot.md` | the 6 lever menu (Class B only) |
| Platform linter output | pasted by user below the trigger phrase |

## Procedure

1. **Classify the block.** Read the linter output. Decide Class A, Class B, or both.

2. **For each Class A complaint:**
   - Re-run Council A targeted on the cited QC dimension. The sub-agent re-greps the specific claim against `_aux/Universe_Split/` and `_aux/Fact_Ledger.json`.
   - If the linter is right: revise `5_Prompt.txt`, then re-run `python Validators/validate.py --phase prompt --task Tasks/<TASK_DIR>` and re-run Council A + B. Log the change to `_aux/Linter_Decision.md` (date, complaint, change made, why).
   - If the linter is wrong: write a justification using the template in `Reference/Linter_Playbook.md`. Two to five sentences, no em-dashes, no references to guides / specs / frameworks, first-person human voice, cites a specific record. Save to `_aux/Linter_Justifications.md`. Append to `Tasks/_meta/Linter_Justifications.md` with the linter excerpt, the justification, and a placeholder for the reviewer's response.

3. **For Class B (similarity ≥ 40%):**
   - Identify which prior task matched (linter output usually names it).
   - Pick the smallest combination from `Similarity_Pivot.md`'s 6 levers that breaks the match while preserving the Hardness Plan levers.
   - Redraft `5_Prompt.txt` from scratch using the new shape. Do NOT surface-edit the old prompt.
   - Re-run `python Validators/validate.py --phase prompt --task Tasks/<TASK_DIR>`, then re-run Council A and Council B (both must GO).
   - Log the pivot to `Tasks/_meta/Similarity_Log.md` (date, linter excerpt, matched-task reference, levers used, outcome placeholder).

4. **If both classes hit:** revise the Class A items first, then run the Class B pivot in the second pass.

5. **Update `_aux/Linter_Decision.md`** with a one-paragraph summary: what the linter blocked, what we did (revised / justified / pivoted), and the final state of `5_Prompt.txt`.

## Exit criteria

- `5_Prompt.txt` is either revised (validator + councils re-cleared) or unchanged with a justification on file.
- `_aux/Linter_Decision.md` exists and documents the resolution.
- For pushbacks: `_aux/Linter_Justifications.md` + `Tasks/_meta/Linter_Justifications.md` updated.
- For pivots: `Tasks/_meta/Similarity_Log.md` updated.

## Next step

User resubmits to the platform. If still blocked, run `PIPELINE S1.5` again. Otherwise proceed to `PIPELINE S2`.

## Bootstrap

Read root `AGENTS.md` first. Justifications follow the strict style in `Reference/Linter_Playbook.md` — concise, human, no em-dashes, no mention of guides or specs.

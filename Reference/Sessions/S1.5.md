# PIPELINE S1.5 — Linter Blocker

Triggered by: `PIPELINE S1.5 — Tasks/<TASK_DIR>` followed by the platform linter output pasted below the line.

## What this phase does

Handles platform linter blocks on the prompt — works for both flows:

- **CB-mode** (default): the CB just submitted a freshly drafted prompt via S1. Fixes land in `5_Prompt.txt` directly.
- **Review-mode**: the candidate's prompt is stuck behind the platform linter, blocking access to the prefilled OE / rubrics. You can either push back with a justification (preferred — preserves the candidate's prompt for rating) or apply a minimal fix that gets past the linter without changing the scenario shape. Fixes land in `_aux/REVIEW_prompt_draft.txt` so the rating original (`5_Prompt.txt`) stays pristine.

Two classes:

- **Class A** — misalignment on a specific QC dimension (business function, persona, data, truthfulness, feasibility, etc.). Decide: revise the prompt, or push back with a concise justification.
- **Class B** — similarity ≥ 40% with a prior submission. Pivot the prompt using `Reference/Similarity_Pivot.md`. No justification path on similarity (the platform ceiling is binding).

## Procedure

1. **Detect mode (CB vs Review).** Two signals:
   - **CB-mode**: presence of `_aux/Universe_Split/` AND absence of candidate-prefilled OE / rubrics (operator authored the prompt themselves via S1).
   - **Review-mode**: presence of candidate-prefilled `6_Oracle_Events.txt` AND `7_Rubrics.json` (regardless of whether `_aux/Council_Reports/REVIEW_triage.md` exists — REVIEW may not have run yet if the linter is blocking the candidate's prompt). In Review-mode, the original `5_Prompt.txt` MUST stay pristine for rating; any fix routes to `_aux/REVIEW_prompt_draft.txt`.

   This detection runs whether REVIEW has been invoked yet or not — Review-mode tasks can hit linter BEFORE REVIEW (linter blocks operator from accessing the prefilled artifacts) or AFTER REVIEW emitted a corrected prompt draft and re-submitted.

2. **Classify the block.** Read the linter output. Decide Class A (misalignment on a QC dimension — feasibility, persona, business function, truthfulness, etc.) or Class B (similarity ≥ 40% with a prior submission) or both.

3. **Skeptical-first decision flow for Class A.** The platform linter has limited intelligence and produces false positives. Invalidation (a justification) is CHEAP — any reasonable justification gets accepted because the platform does not intelligently evaluate justifications. Revision is EXPENSIVE — re-runs validator + Council A + B + AUDIT, plus risks introducing new defects.

   Default assumption: the linter MAY be wrong. Treat the linter as the prosecution; its claim must be supported by per-task universe evidence to warrant revision.

   For each Class A complaint, run this decision flow IN ORDER:

   1. **Score skeptically against the per-task universe.** Spawn Council A targeted re-grep on the cited QC dimension against `_aux/Universe_Split/` and `_aux/Fact_Ledger.json`.

   2. **Classify the linter:**
      - **Clearly wrong** (universe grep contradicts the linter): **INVALIDATE with justification** (preferred). Write per the Class A template in `Reference/Linter_Playbook.md`, save to `_aux/Linter_Justifications.md`. Run `python Validators/check_justification.py Tasks/<TASK_DIR>/_aux/Linter_Justifications.md` — exit 0 required, revise + re-run if not. Append to `Tasks/_meta/Linter_Justifications.md` with the linter excerpt, the justification, and a placeholder for the reviewer's response.
      - **Ambiguous** (universe grep is mixed, the linter's claim is subjective, or the linter cites a soft pattern): **INVALIDATE with justification** (cost asymmetry favors invalidation — platform doesn't intelligently evaluate). Same path as clearly-wrong.
      - **Clearly right** (universe grep unambiguously confirms the gap): **REVISE** (high cost). For CB-mode, revise `5_Prompt.txt` in place; then re-run `python Validators/validate.py --phase prompt --task Tasks/<TASK_DIR>` and re-run Council A + B + AUDIT (per the v7 auto-fire in S1 step 8). For Review-mode, write the fix to `_aux/REVIEW_prompt_draft.txt` (create or update); add a Pending row to `changes.md`; re-run validator on a temp-copy of the draft renamed `5_Prompt.txt` in a scratch dir, then re-run Council A + B + AUDIT against the draft. Originals stay untouched.

   The DEFAULT path is invalidation. Choose revision only when the universe grep gives you no plausible counter to the linter's claim.

4. **Skeptical-first decision flow for Class B (similarity ≥ 40% per platform linter).**

   First, run the multi-dimensional similarity scorer to estimate the probable COMPOSITE similarity across lexical, token, word-count, persona, business function, and universe dimensions:
   ```
   python Validators/calc_similarity.py Tasks/<TASK_DIR>
   python Validators/calc_similarity.py Tasks/<TASK_DIR> --explain <matched_prior_task_dir>
   ```
   The first command emits the full per-dimension report. The second prints the per-dimension breakdown vs the specific prior task the platform linter named. Same skeptical-first principle: any reasonable justification gets accepted by the platform, so invalidation is the cheap default whenever the case is plausible.

   **Decision flow (simplified, two bands):**

   1. **Composite probable similarity < 40** → **INVALIDATE with justification** (move on). The multiplicative context-weighted composite has already accounted for any differentiators (BF differs × 0.6, persona differs × 0.6, universe differs × 0.7) — a sub-40 composite means either the lexical overlap is genuinely low OR the contextual differentiators are strong enough that invalidation is defensible. Write per the Class B similarity-invalidation template in `Reference/Linter_Playbook.md` — 2-3 sentences, plain humanlike voice, cite the differentiating dimensions from the `--explain` output, no em-dashes, no internal terminologies, no stats. Save to `_aux/Linter_Justifications.md`. Run `python Validators/check_justification.py` — exit 0 required. Append to `Tasks/_meta/Linter_Justifications.md`.

   2. **Composite probable similarity ≥ 40** → **PIVOT**. The composite is high even after the contextual multiplier was applied — meaning constants ALIGN with the matched prior task (no differentiators to lean on) AND the lexical overlap is substantial. Structural similarity is genuine, not contextual. Pick the smallest combination from `Reference/Similarity_Pivot.md`'s 6 levers that breaks the match while preserving the Hardness Plan levers. CB-mode: redraft `5_Prompt.txt` from scratch using the new shape. Review-mode: redraft to `_aux/REVIEW_prompt_draft.txt`. Re-run validator + Council A + B + AUDIT (per S1 step 8 auto-fire).

   Log to `Tasks/_meta/Similarity_Log.md` (date, linter excerpt, matched-task reference, composite + per-dim scores, action taken — invalidate / pivot, outcome placeholder).

5. **If both classes hit:** resolve the Class B (similarity) first via its decision flow, then resolve any remaining Class A items.

6. **Update `_aux/Linter_Decision.md`** with a one-paragraph summary: what the linter blocked, what we did (invalidated / revised / pivoted), the skeptical-first reasoning that drove the choice, and the final state of `5_Prompt.txt` (or `_aux/REVIEW_prompt_draft.txt` in Review-mode).

7. **Run the justification voice gate** before declaring the phase complete. Any reviewer-facing justification file must pass `check_justification.py` with exit 0:
   ```
   python Validators/check_justification.py Tasks/<TASK_DIR>/_aux/Linter_Justifications.md
   ```
   Skip this step only if no pushback was authored (revise-only or pivot-only resolution). If the file exists and the script returns non-zero, revise the justification (strip every forbidden term per the per-hit report) and re-run until clean. Do NOT ship a pushback that fails the voice gate — process leakage to the platform reviewer immediately undermines the pushback.

8. **Strict veteran audit on any revised prompt (MANDATORY, auto-fire when prompt was modified).** If this S1.5 pass revised `5_Prompt.txt` (CB mode) or `_aux/REVIEW_prompt_draft.txt` (Review mode) — i.e., a Class A revise OR a Class B pivot path was taken — spawn an audit sub-agent (`oracle`) using the prompt template in `Reference/Sessions/AUDIT.md` with `--phase prompt` configured against the revised prompt file. Save the report to `_aux/Council_Reports/AUDIT_prompt.md` (overwrite any prior S1 audit since the prompt has changed).

   Skip this step ONLY if the resolution was justification-only (no prompt edit). Verdict handling matches S1 step 8: `PASS (STRICT)` → proceed to exit; `REVISE` → apply fixes + re-run validator + Council A + Council B + AUDIT (iteration cap 3); `REBUILD` → STOP and recommend `PIPELINE REDO`. The rationale is identical to S1: catching defects at the producing phase is cheaper than catching them at FINAL or at platform-reviewer time.

## Exit criteria

- `5_Prompt.txt` is either revised (validator + councils re-cleared) or unchanged with a justification on file.
- `_aux/Linter_Decision.md` exists and documents the resolution.
- For pushbacks: `_aux/Linter_Justifications.md` + `Tasks/_meta/Linter_Justifications.md` updated AND `python Validators/check_justification.py Tasks/<TASK_DIR>/_aux/Linter_Justifications.md` exits 0.
- For pivots: `Tasks/_meta/Similarity_Log.md` updated.
- For any revised prompt (Class A revise OR Class B pivot): `_aux/Council_Reports/AUDIT_prompt.md` returns `PASS (STRICT)`. `REVISE` iterations must resolve within the 3-round cap; `REBUILD` STOPs to REDO.

## STOP gate

This phase ends here. End your response. The user resubmits the revised prompt (or the original prompt + justification) to the platform.

Two next-trigger paths:
- Platform still blocks → `PIPELINE S1.5 — Tasks/<TASK_DIR>` (paste the new linter output) in a fresh chat.
- Platform clears → `PIPELINE S2 — Tasks/<TASK_DIR>` in a fresh chat.

Do NOT proceed to OE drafting in this chat.

**If the user pastes follow-up content in this chat** (a new linter response from a later platform attempt, the OE phase inputs, fix attempts on a different deliverable, or an unrelated question), do NOT process it. Reply: "This chat is single-shot for S1.5 linter handling on the prompt revision just resolved. Please open a fresh chat and invoke the appropriate next trigger from the dispatch table." Chaining inside one chat defeats the entire pipeline.

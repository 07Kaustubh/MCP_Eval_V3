# PIPELINE S1.5 — Linter Blocker

Triggered by: `PIPELINE S1.5 — Tasks/<TASK_DIR>` followed by the platform linter output pasted below the line.

## What this phase does

Handles platform linter blocks on the prompt — works for both flows:

- **CB-mode** (default): the CB just submitted a freshly drafted prompt via S1. Fixes land in `5_Prompt.txt` directly.
- **Review-mode**: the candidate's prompt is stuck behind the platform linter, blocking access to the prefilled OE / rubrics. You can either push back with a justification (preferred — preserves the candidate's prompt for rating) or apply a minimal fix that gets past the linter without changing the scenario shape. Fixes land in `_aux/REVIEW_prompt_draft.txt` so the rating original (`5_Prompt.txt`) stays pristine.

Two classes:

- **Class A** — misalignment on a specific QC dimension (business function, persona, data, truthfulness, feasibility, etc.). Decide: revise the prompt, or push back with a concise justification.
- **Class B** — similarity ≥ 40% with a prior submission. Pivot the prompt using `Reference/Similarity_Pivot.md`. No justification path on similarity (the platform ceiling is binding).

## Required inputs

| File | Source |
|---|---|
| `Tasks/<TASK_DIR>/5_Prompt.txt` | S1 (CB) OR candidate-original (Review) |
| `Tasks/<TASK_DIR>/_aux/Fact_Ledger.json` | S0; used by the validator and Council A to enforce groundedness on any revised prompt — same gate as S1 / S3. |
| `Tasks/<TASK_DIR>/_aux/Council_Reports/REVIEW_triage.md` | OPTIONAL — presence indicates Review-mode (fixes route to scratch draft, not in-place) |
| `Tasks/<TASK_DIR>/_aux/Universe_Split/*` | S0 |
| `Tasks/<TASK_DIR>/_aux/Universe_Index/*` | S0 |
| `Tasks/<TASK_DIR>/_aux/Hardness_Plan.md` | HARDNESS |
| `Reference/Linter_Playbook.md` | decision flow + justification template |
| `Reference/Similarity_Pivot.md` | the 6 lever menu (Class B only) |
| Platform linter output | pasted by user below the trigger phrase |

## Phase-readiness gate (run FIRST)

```
python Validators/phase_ready.py --phase s1.5 --task Tasks/<TASK_DIR>
```

Refuses if upstream artifacts are missing. If it STOPs, run the upstream phase first.

## Procedure

1. **Classify the block.** Read the linter output. Decide Class A, Class B, or both.

2. **For each Class A complaint:**
   - Re-run Council A targeted on the cited QC dimension. The sub-agent re-greps the specific claim against `_aux/Universe_Split/` and `_aux/Fact_Ledger.json` (same groundedness gate as S1 / S3).
   - **CB-mode** (no `_aux/Council_Reports/REVIEW_triage.md` present, or in-place fix is wanted): if the linter is right, revise `5_Prompt.txt` in place; then re-run `python Validators/validate.py --phase prompt --task Tasks/<TASK_DIR>` (the validator does the full grounded sweep via Fact_Ledger) and re-run Council A + B. Log the change to `_aux/Linter_Decision.md`.
   - **Review-mode** (a REVIEW already ran and `5_Prompt.txt` must stay pristine for rating): if the linter is right, write the fix to `_aux/REVIEW_prompt_draft.txt` (or update it if it already exists from a prior REVIEW pass). Add a Pending row to `changes.md` describing the linter complaint + the fix. Re-run the validator on a temp-copy of the draft renamed `5_Prompt.txt` in a scratch dir; then re-run Council A + B against the draft. Originals stay untouched.
   - If the linter is wrong (either mode): write a justification using the template in `Reference/Linter_Playbook.md`. Two to five sentences, no em-dashes, no references to guides / specs / frameworks, first-person human voice, cites a specific record. Save to `_aux/Linter_Justifications.md`. Append to `Tasks/_meta/Linter_Justifications.md` with the linter excerpt, the justification, and a placeholder for the reviewer's response.

3. **For Class B (similarity ≥ 40%):**
   - Identify which prior task matched (linter output usually names it).
   - Pick the smallest combination from `Similarity_Pivot.md`'s 6 levers that breaks the match while preserving the Hardness Plan levers.
   - **CB-mode**: redraft `5_Prompt.txt` from scratch using the new shape. Do NOT surface-edit the old prompt.
   - **Review-mode**: redraft the corrected prompt into `_aux/REVIEW_prompt_draft.txt` and add a Pending row to `changes.md`. Do NOT touch `5_Prompt.txt`.
   - Re-run `python Validators/validate.py --phase prompt --task Tasks/<TASK_DIR>` (against the in-place file or a temp-copy of the draft), then re-run Council A and Council B (both must GO).
   - Log the pivot to `Tasks/_meta/Similarity_Log.md` (date, linter excerpt, matched-task reference, levers used, outcome placeholder).

4. **If both classes hit:** revise the Class A items first, then run the Class B pivot in the second pass.

5. **Update `_aux/Linter_Decision.md`** with a one-paragraph summary: what the linter blocked, what we did (revised / justified / pivoted), and the final state of `5_Prompt.txt`.

6. **Run the justification voice gate** before declaring the phase complete. Any reviewer-facing justification file must pass `check_justification.py` with exit 0:
   ```
   python Validators/check_justification.py Tasks/<TASK_DIR>/_aux/Linter_Justifications.md
   ```
   Skip this step only if no pushback was authored (revise-only or pivot-only resolution). If the file exists and the script returns non-zero, revise the justification (strip every forbidden term per the per-hit report) and re-run until clean. Do NOT ship a pushback that fails the voice gate — process leakage to the platform reviewer immediately undermines the pushback.

7. **Strict veteran audit on any revised prompt (MANDATORY, auto-fire when prompt was modified).** If this S1.5 pass revised `5_Prompt.txt` (CB mode) or `_aux/REVIEW_prompt_draft.txt` (Review mode) — i.e., a Class A revise OR a Class B pivot path was taken — spawn an audit sub-agent (`oracle`) using the prompt template in `Reference/Sessions/AUDIT.md` with `--phase prompt` configured against the revised prompt file. Save the report to `_aux/Council_Reports/AUDIT_prompt.md` (overwrite any prior S1 audit since the prompt has changed).

   Skip this step ONLY if the resolution was justification-only (no prompt edit). Verdict handling matches S1 step 8: `PASS (STRICT)` → proceed to exit; `REVISE` → apply fixes + re-run validator + Council A + Council B + AUDIT (iteration cap 3); `REBUILD` → STOP and recommend `PIPELINE REDO`. The rationale is identical to S1: catching defects at the producing phase is cheaper than catching them at FINAL or at platform-reviewer time.

## Exit criteria

- `5_Prompt.txt` is either revised (validator + councils re-cleared) or unchanged with a justification on file.
- `_aux/Linter_Decision.md` exists and documents the resolution.
- For pushbacks: `_aux/Linter_Justifications.md` + `Tasks/_meta/Linter_Justifications.md` updated AND `python Validators/check_justification.py Tasks/<TASK_DIR>/_aux/Linter_Justifications.md` exits 0.
- For pivots: `Tasks/_meta/Similarity_Log.md` updated.
- For any revised prompt (Class A revise OR Class B pivot): `_aux/Council_Reports/AUDIT_prompt.md` returns `PASS (STRICT)`. `REVISE` iterations must resolve within the 3-round cap; `REBUILD` STOPs to REDO.

## STOP gate

This phase ends here. End your response. The user resubmits the revised prompt (or the original prompt + justification) to the platform.

Two next-trigger paths:
- Platform still blocks → `PIPELINE S1.5 — Tasks/<TASK_DIR>` (paste the new linter output) in a fresh chat.
- Platform clears → `PIPELINE S2 — Tasks/<TASK_DIR>` in a fresh chat.

Do NOT proceed to OE drafting in this chat.

**If the user pastes follow-up content in this chat** (a new linter response from a later platform attempt, the OE phase inputs, fix attempts on a different deliverable, or an unrelated question), do NOT process it. Reply: "This chat is single-shot for S1.5 linter handling on the prompt revision just resolved. Please open a fresh chat and invoke the appropriate next trigger from the dispatch table." Chaining inside one chat defeats the entire pipeline.

## Bootstrap

Read root `AGENTS.md` first. Justifications follow the strict style in `Reference/Linter_Playbook.md` — concise, human, no em-dashes, no mention of guides or specs.

# PIPELINE S1.5 — Linter Blocker

Triggered by: `PIPELINE S1.5 — <TASK_DIR>` followed by the platform linter output pasted below the line.

## What this phase does

Handles platform linter blocks on the prompt — works for both flows:

- **CB-mode** (default): the CB just submitted a freshly drafted prompt via S1. Fixes land in `5_Prompt.txt` directly.
- **Review-mode**: the candidate's prompt is stuck behind the platform linter, blocking access to the prefilled OE / rubrics. You can either push back with a justification (preferred — preserves the candidate's prompt for rating) or apply a minimal fix that gets past the linter without changing the scenario shape. Fixes land in `_aux/REVIEW_prompt_draft.txt` so the rating original (`5_Prompt.txt`) stays pristine.

Two classes:

- **Class A** — misalignment on a specific QC dimension (business function, persona, data, truthfulness, feasibility, etc.). Decide: revise the prompt, or push back with a concise justification.
- **Class B** — similarity ≥ 40% with a prior submission. Pivot the prompt using `Reference/Similarity_Pivot.md`. No justification path on similarity (the platform ceiling is binding).

## Phase-readiness gate (run FIRST)

```
python Validators/phase_ready.py --phase s1.5 --task <TASK_DIR>
```

Refuses if upstream artifacts are missing. If it STOPs, run the upstream phase first.

## Procedure

1. **Classify the block.** Read the linter output. Decide Class A, Class B, or both.

2. **For each Class A complaint:**
   - **Re-run the right council for the flag type.** If the linter cited a grounding claim (a specific identifier, amount, persona email, relationship), re-run Council A on that claim — the sub-agent re-greps against `_aux/Universe_Split/` and `_aux/Fact_Ledger.json` (same groundedness gate as S1 / S3). If the linter cited a QC sub-dim (Business Function, Persona fit, Feasibility, Clarity, Truthfulness, etc.), re-run Council B targeted on that sub-dim — the council must produce an explicit Pass-band score with the per-task universe evidence the justification will name. **If the cited evidence does not satisfy the Pass-band wording verbatim from `Docs/7_QC_Spec_Doc1.json`, the pushback is invalid and the prompt must be revised, not justified.** A common failure mode: defending a Business Function flag with a Persona-fit argument — Persona and Business Function are independent sub-dims and the persona-fit evidence does NOT satisfy the BF Pass band ("scenario within the assigned function").
   - **CB-mode** (no `_aux/Council_Reports/REVIEW_triage.md` present, or in-place fix is wanted): if the linter is right, revise `5_Prompt.txt` in place; then re-run `python Validators/validate.py --phase prompt --task <TASK_DIR>` (the validator does the full grounded sweep via Fact_Ledger) and re-run Council A + B. Log the change to `_aux/Linter_Decision.md`.
   - **Review-mode** (a REVIEW already ran and `5_Prompt.txt` must stay pristine for rating): if the linter is right, write the fix to `_aux/REVIEW_prompt_draft.txt` (or update it if it already exists from a prior REVIEW pass). Add a Pending row to `changes.md` describing the linter complaint + the fix. Re-run the validator on a temp-copy of the draft renamed `5_Prompt.txt` in a scratch dir; then re-run Council A + B against the draft. Originals stay untouched.
   - If the linter is wrong (either mode): write a justification using the template in `Reference/Linter_Playbook.md`. Two to five sentences, no em-dashes, no references to guides / specs / frameworks, first-person human voice, cites a specific record. Save to `_aux/Linter_Justifications.md`. Append to `Submitted-Tasks/_meta/Linter_Justifications.md` with the linter excerpt, the justification, and a placeholder for the reviewer's response.

3. **For Class B (similarity ≥ 40%):**
   - Identify which prior task matched (linter output usually names it).
   - Pick the smallest combination from `Similarity_Pivot.md`'s 6 levers that breaks the match while preserving the Hardness Plan levers.
   - **CB-mode**: redraft `5_Prompt.txt` from scratch using the new shape. Do NOT surface-edit the old prompt.
   - **Review-mode**: redraft the corrected prompt into `_aux/REVIEW_prompt_draft.txt` and add a Pending row to `changes.md`. Do NOT touch `5_Prompt.txt`.
   - Re-run `python Validators/validate.py --phase prompt --task <TASK_DIR>` (against the in-place file or a temp-copy of the draft), then re-run Council A and Council B (both must GO).
   - Log the pivot to `Submitted-Tasks/_meta/Similarity_Log.md` (date, linter excerpt, matched-task reference, levers used, outcome placeholder).

4. **If both classes hit:** revise the Class A items first, then run the Class B pivot in the second pass.

5. **Update `_aux/Linter_Decision.md`** with a one-paragraph summary: what the linter blocked, what we did (revised / justified / pivoted), and the final state of `5_Prompt.txt`.

## Exit criteria

- `5_Prompt.txt` is either revised (validator + councils re-cleared) or unchanged with a justification on file.
- `_aux/Linter_Decision.md` exists and documents the resolution.
- For pushbacks: `_aux/Linter_Justifications.md` + `Submitted-Tasks/_meta/Linter_Justifications.md` updated.
- For pivots: `Submitted-Tasks/_meta/Similarity_Log.md` updated.

## STOP gate

This phase ends here. End your response. The user resubmits the revised prompt (or the original prompt + justification) to the platform.

Two next-trigger paths:
- Platform still blocks → `PIPELINE S1.5 — <TASK_DIR>` (paste the new linter output) in a fresh chat.
- Platform clears → `PIPELINE S2 — <TASK_DIR>` in a fresh chat.

Do NOT proceed to OE drafting in this chat.
